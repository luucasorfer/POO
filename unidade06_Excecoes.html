1. O que é uma Exceção?

Uma exceção é um evento inesperado que ocorre durante a execução de um programa e interrompe seu fluxo normal. Em vez de deixar o programa travar de forma abrupta, Java oferece um mecanismo para capturar e tratar esses erros de maneira controlada.

Definição: Exceção é uma condição anormal que surge durante a execução de um programa. O Java usa objetos para representar essas condições e permite que o código as trate sem encerrar o programa de forma inesperada.

O que acontece sem tratamento

Sem tratamento de exceções, um erro simples como dividir por zero ou acessar um índice inexistente encerra o programa imediatamente e exibe uma mensagem de erro no console:

          

             Sem tratamento — programa encerra abruptamente
            

              

              

              

            

          

          public class Main {
    public static void main(String[] args) {
        int resultado = 10 / 0;  // ❌ divisão por zero
        System.out.println(resultado);  // nunca chega aqui
    }
}

Saída — erro no console:

Exception in thread "main" java.lang.ArithmeticException: / by zero
    at Main.main(Main.java:3)

Tipos de erro em Java

Java distingue três categorias de problemas:

E
Error — erros graves da JVM

Problemas que o programa não pode tratar: falta de memória (OutOfMemoryError), estouro de pilha (StackOverflowError). O programa geralmente não consegue se recuperar.

R
RuntimeException — exceções não verificadas

Ocorrem em tempo de execução por bugs no código: divisão por zero, índice inválido, referência nula. O compilador não obriga o tratamento.

C
Checked Exceptions — exceções verificadas

Condições previsíveis que podem ocorrer: arquivo não encontrado, falha de rede. O compilador exige que sejam tratadas ou declaradas.

2. Hierarquia de Exceções

Em Java, todas as exceções são classes que herdam de Throwable. Conhecer essa hierarquia é fundamental para saber qual tipo capturar:

Throwable
├─ Error OutOfMemoryError, StackOverflowError
└─ Exception
├─ RuntimeException (não verificadas)
├─ NullPointerException
├─ ArithmeticException
├─ ArrayIndexOutOfBoundsException
└─ NumberFormatException
├─ IOException (verificadas)
└─ FileNotFoundException
└─ SuaExcecaoCustom (criada por você)

Exceções mais comuns

Exceção Tipo Causa comum

3. try / catch / finally

O mecanismo principal de tratamento de exceções em Java usa três blocos: try, catch e finally.

try
Bloco protegido Contém o código que pode lançar uma exceção. Se uma exceção ocorrer aqui, a execução é interrompida e vai direto para o catch.
catch
Captura e trata a exceção Executado somente se a exceção declarada ocorrer no try. Pode haver múltiplos catch para tipos diferentes.
finally
Sempre executado — opcional Roda independentemente de ter ocorrido exceção ou não. Ideal para liberar recursos (fechar arquivos, conexões, etc.).

Estrutura básica

          

             try / catch — estrutura básica
            

              

              

              

            

          

          public class Main {
    public static void main(String[] args) {

        try {
            int resultado = 10 / 0;  // lança ArithmeticException
            System.out.println(resultado);  // não executa
        } catch (ArithmeticException e) {
            System.out.println("Erro: divisão por zero!");
            System.out.println(e.getMessage());  // detalhe do erro
        }

        System.out.println("Programa continua normalmente.");
    }
}

Saída:

Erro: divisão por zero!
/ by zero
Programa continua normalmente.

O bloco finally

O finally é executado sempre — seja com exceção ou sem. É o lugar certo para liberar recursos:

          

             try / catch / finally
            

              

              

              

            

          

          try {
    System.out.println("Tentando executar...");
    int x = 5 / 0;                 // lança exceção aqui
} catch (ArithmeticException e) {
    System.out.println("Exceção capturada: " + e.getMessage());
} finally {
    System.out.println("finally: sempre executa!");
}

Saída:

Tentando executar...
Exceção capturada: / by zero
finally: sempre executa!

Múltiplos catch

Você pode encadear vários catch para tratar diferentes tipos de exceção de formas distintas. O Java verifica cada catch em ordem e executa apenas o primeiro que corresponder:

          

             Múltiplos catch — tratamentos específicos
            

              

              

              

            

          

          import java.util.Scanner;

Scanner sc = new Scanner(System.in);

try {
    System.out.print("Digite um número: ");
    int numero = Integer.parseInt(sc.nextLine());

    int[] vetor = {10, 20, 30};
    System.out.println(vetor[numero]);

} catch (NumberFormatException e) {
    // usuário digitou texto ao invés de número
    System.out.println("Entrada inválida — digite apenas números.");

} catch (ArrayIndexOutOfBoundsException e) {
    // índice fora do tamanho do vetor
    System.out.println("Índice inválido — use 0, 1 ou 2.");

} catch (Exception e) {
    // captura qualquer outra exceção não prevista
    System.out.println("Erro inesperado: " + e.getMessage());
}

Regra importante: sempre coloque os catch do tipo mais específico para o mais genérico. Se Exception vier primeiro, ele capturará tudo e os outros catch jamais serão executados — o compilador acusará erro.

Multi-catch — um catch, vários tipos

Desde o Java 7, você pode capturar múltiplos tipos em um único catch usando |:

          

             Multi-catch (Java 7+)
            

              

              

              

            

          

          try {
    // código que pode lançar diferentes exceções
} catch (NumberFormatException | ArrayIndexOutOfBoundsException e) {
    System.out.println("Erro de formato ou índice: " + e.getMessage());
}

4. throw e throws

Além de capturar exceções, você pode lançá-las manualmente com throw, e declarar que um método pode lançar exceções com throws.

throw

Lança uma exceção explicitamente dentro do código.

throws

Declara na assinatura do método que ele pode lançar uma exceção.

Usando throw

          

             Lançando uma exceção com throw
            

              

              

              

            

          

          static void setIdade(int idade) {
    if (idade < 0) {
        throw new IllegalArgumentException(
            "Idade inválida: " + idade + ". Deve ser >= 0."
        );
    }
    System.out.println("Idade definida: " + idade);
}

public static void main(String[] args) {
    try {
        setIdade(25);   // OK → Idade definida: 25
        setIdade(-5);   // lança IllegalArgumentException
    } catch (IllegalArgumentException e) {
        System.out.println("Capturado: " + e.getMessage());
    }
}

Usando throws — propagando exceções

Quando um método não quer tratar a exceção localmente, ele pode propagar para quem o chamou usando throws na assinatura:

          

             throws — declaração na assinatura do método
            

              

              

              

            

          

          import java.io.*;

// Método declara que PODE lançar IOException
static void lerArquivo(String caminho) throws IOException {
    FileReader leitor = new FileReader(caminho);
    // quem chamar lerArquivo() é responsável por tratar a exceção
}

public static void main(String[] args) {
    try {
        lerArquivo("dados.txt");
    } catch (IOException e) {
        System.out.println("Arquivo não encontrado: " + e.getMessage());
    }
}

throw vs throws em resumo:

  • throw — ação: lança um objeto de exceção
  • throws — declaração: avisa que o método pode lançar uma exceção

5. Exceções Personalizadas

Você pode criar suas próprias exceções para representar erros específicos do seu domínio de negócio. Para isso, basta criar uma classe que herda de Exception (checked) ou de RuntimeException (unchecked).

Quando criar uma exceção personalizada?

  • Quando nenhuma exceção padrão do Java descreve bem o erro do seu sistema
  • Quando você quer fornecer uma mensagem mais clara e específica
  • Quando quer separar erros de negócio de erros técnicos

Criando uma exceção personalizada

          

             SaldoInsuficienteException.java
            

              

              

              

            

          

          // Herda de Exception (checked) — obriga o tratamento
public class SaldoInsuficienteException extends Exception {

    private double saldoDisponivel;
    private double valorSolicitado;

    public SaldoInsuficienteException(double saldo, double valor) {
        super("Saldo insuficiente. Disponível: R$ " + saldo
              + ", Solicitado: R$ " + valor);
        this.saldoDisponivel = saldo;
        this.valorSolicitado  = valor;
    }

    public double getSaldoDisponivel() { return saldoDisponivel; }
    public double getValorSolicitado()  { return valorSolicitado; }

}

Usando a exceção personalizada

          

             ContaBancaria.java — usando a exceção customizada
            

              

              

              

            

          

          public class ContaBancaria {

    private double saldo;

    public ContaBancaria(double saldo) {
        this.saldo = saldo;
    }

    public void sacar(double valor) throws SaldoInsuficienteException {
        if (valor > saldo) {
            throw new SaldoInsuficienteException(saldo, valor);
        }
        saldo -= valor;
        System.out.println("Saque de R$ " + valor + " realizado.");
    }

}

// Uso na Main:
ContaBancaria conta = new ContaBancaria(500.00);

try {
    conta.sacar(200.00);  // OK → Saque de R$ 200.0 realizado.
    conta.sacar(400.00);  // lança SaldoInsuficienteException
} catch (SaldoInsuficienteException e) {
    System.out.println(e.getMessage());
    System.out.println("Disponível: R$ " + e.getSaldoDisponivel());
}

Saída:

Saque de R$ 200.0 realizado.
Saldo insuficiente. Disponível: R$ 300.0, Solicitado: R$ 400.0
Disponível: R$ 300.0

Exceção unchecked personalizada

Se quiser que o compilador não obrigue o tratamento, herde de RuntimeException:

          

             Exceção unchecked — herda de RuntimeException
            

              

              

              

            

          

          // Não obriga o uso de throws / try-catch
public class IdadeInvalidaException extends RuntimeException {

    public IdadeInvalidaException(int idade) {
        super("Idade inválida: " + idade + ". Deve ser entre 0 e 150.");
    }

}

6. Métodos Úteis de Exception

Todo objeto de exceção em Java herda métodos que ajudam a entender e registrar o que aconteceu:

          

             Métodos do objeto de exceção
            

              

              

              

            

          

          try {
    int x = 10 / 0;

} catch (ArithmeticException e) {

    e.getMessage();
    // → "/ by zero"   — mensagem resumida do erro

    e.toString();
    // → "java.lang.ArithmeticException: / by zero"

    e.printStackTrace();
    // → imprime a pilha completa de chamadas no console
    //   útil para depuração, mas evite em produção

    e.getClass().getName();
    // → "java.lang.ArithmeticException" — nome completo da classe

}

getClass().getName()StringNome completo da classe da exceção

7. Boas Práticas

Capture exceções específicas, não genéricas Prefira catch (NumberFormatException e) a catch (Exception e). Capturar tudo esconde bugs e dificulta a depuração.
Nunca deixe o catch vazio Um catch (Exception e) { } vazio engole o erro silenciosamente — o pior tipo de bug. Sempre registre ou trate a exceção.
Use finally para liberar recursos Fechar conexões, arquivos e streams no finally garante que isso aconteça mesmo se uma exceção ocorrer.
Mensagens de exceção devem ser úteis Ao criar exceções, forneça mensagens claras e informativas. "Idade inválida: -3. Deve ser >= 0" é muito melhor que apenas "erro".
Não use exceções para controle de fluxo Exceções são para situações excepcionais, não para lógica normal. Usar try/catch no lugar de if/else torna o código lento e confuso.
Registre as exceções em produção Use um logger (java.util.logging ou bibliotecas como Log4j) em vez de printStackTrace() para registrar erros em sistemas reais.

Aplicando as boas práticas

❌ Ruim
              

                
                

                  

                  

                  

                

              

              try {
    processarDados();
} catch (Exception e) {
    // catch vazio — erro sumiu!
}
✅ Bom
              

                
                

                  

                  

                  

                

              

              try {
    processarDados();
} catch (IOException e) {
    System.err.println(
        "Falha na leitura: "
        + e.getMessage());
}

8. Exercícios Práticos

01
Divisão Segura com try/catch
Nível 1

Crie um método dividir(int a, int b) que retorne o resultado da divisão. Use try/catch para tratar ArithmeticException e retorne 0 em caso de divisão por zero, exibindo uma mensagem clara.

                

                   Solução
                  

                    

                    

                    

                  

                

                static int dividir(int a, int b) {
    try {
        return a / b;
    } catch (ArithmeticException e) {
        System.out.println("Erro: não é possível dividir por zero.");
        return 0;
    }
}

// Teste
System.out.println(dividir(10, 2));   // 5
System.out.println(dividir(10, 0));   // Erro: não é possível dividir por zero. → 0
02
Leitura de Número com Validação
Nível 1

Crie um método lerInteiro(String texto) que tente converter uma String em inteiro. Se a conversão falhar por NumberFormatException, retorne -1 e exiba uma mensagem informativa.

                

                   Solução
                  

                    

                    

                    

                  

                

                static int lerInteiro(String texto) {
    try {
        return Integer.parseInt(texto);
    } catch (NumberFormatException e) {
        System.out.println("\"" + texto
            + "\" não é um número inteiro válido.");
        return -1;
    }
}

// Teste
System.out.println(lerInteiro("42"));    // 42
System.out.println(lerInteiro("abc"));   // "abc" não é um número... → -1
System.out.println(lerInteiro("3.14"));  // "3.14" não é um número... → -1
03
Múltiplos catch — Acesso a Array
Nível 2

Crie um método acessarElemento(String[] vetor, String indiceTexto) que converta o texto em índice e retorne o elemento. Trate com catch separados: NumberFormatException (índice não é número) e ArrayIndexOutOfBoundsException (índice fora do vetor). Use finally para sempre exibir "Operação finalizada."

                

                   Solução
                  

                    

                    

                    

                  

                

                static String acessarElemento(String[] vetor, String indiceTexto) {
    try {
        int i = Integer.parseInt(indiceTexto);
        return vetor[i];

    } catch (NumberFormatException e) {
        System.out.println("Índice inválido — não é um número: "
            + indiceTexto);
        return null;

    } catch (ArrayIndexOutOfBoundsException e) {
        System.out.println("Índice fora do intervalo (0 a "
            + (vetor.length - 1) + "): "
            + indiceTexto);
        return null;

    } finally {
        System.out.println("Operação finalizada.");
    }
}

// Teste
String[] frutas = {"maçã", "banana", "laranja"};
System.out.println(acessarElemento(frutas, "1"));    // banana
System.out.println(acessarElemento(frutas, "abc"));  // índice não é número
System.out.println(acessarElemento(frutas, "10"));  // índice fora
04
Sistema de Cadastro com Exceção Personalizada
Nível 3

Crie uma exceção personalizada CpfInvalidoException e uma classe Pessoa que lance essa exceção ao receber um CPF com comprimento diferente de 11 dígitos. Trate a exceção na main.

                

                   Solução
                  

                    

                    

                    

                  

                

                // ── CpfInvalidoException.java ───────────────────────
public class CpfInvalidoException extends Exception {

    public CpfInvalidoException(String cpf) {
        super("CPF inválido: \"" + cpf
              + "\". Deve ter exatamente 11 dígitos.");
    }

}

// ── Pessoa.java ─────────────────────────────────────
public class Pessoa {

    private String nome;
    private String cpf;

    public Pessoa(String nome, String cpf)
            throws CpfInvalidoException {

        if (cpf == null || cpf.length() != 11) {
            throw new CpfInvalidoException(cpf);
        }
        this.nome = nome;
        this.cpf  = cpf;
    }

    public String toString() {
        return nome + " — CPF: " + cpf;
    }

}

// ── Main.java ────────────────────────────────────────
public class Main {
    public static void main(String[] args) {

        try {
            Pessoa p1 = new Pessoa("Lucas", "12345678901");
            System.out.println("Cadastrado: " + p1);

            Pessoa p2 = new Pessoa("Ana", "123");  // lança!
            System.out.println("Cadastrado: " + p2);  // não executa

        } catch (CpfInvalidoException e) {
            System.out.println("Erro de cadastro: " + e.getMessage());
        }

    }
}
              
Cadastrado: Lucas — CPF: 12345678901
Erro de cadastro: CPF inválido: "123". Deve ter exatamente 11 dígitos.

Resumo da Unidade

O que você aprendeu nesta unidade:

  • Uma exceção é um evento inesperado que interrompe o fluxo normal do programa — Java permite capturá-la e tratá-la de forma controlada.
  • A hierarquia parte de Throwable → Error / Exception → RuntimeException (unchecked) ou checked.
  • try protege o código; catch captura a exceção; finally sempre executa.
  • Vários catch devem ir do mais específico para o mais genérico. O multi-catch com | simplifica quando o tratamento é igual.
  • throw lança uma exceção explicitamente; throws declara na assinatura que o método pode lançá-la.
  • Exceções personalizadas estendem Exception (checked) ou RuntimeException (unchecked) e permitem erros mais expressivos.
  • Boas práticas: nunca deixar catch vazio, capturar tipos específicos, usar finally para liberar recursos e fornecer mensagens claras.
