O que é Encapsulamento?

O encapsulamento consiste em proteger os atributos de uma classe, permitindo acesso a eles apenas de forma controlada. Em vez de acessar diretamente uma variável, usamos métodos.

Definição: Esconder os detalhes internos de um objeto e permitir acesso controlado aos seus dados.

Ou seja:

  • Os atributos ficam protegidos
  • O acesso é feito somente por métodos definidos pela classe

O problema sem encapsulamento

Sem encapsulamento, qualquer parte do sistema pode alterar os dados diretamente e de forma inválida: 

          

             Sem encapsulamento — problema
            

              

              

              

            

          

          public class Pessoa {
    public int idade;   // ⚠️ atributo público — qualquer um pode alterar
}

public class Main {
    public static void main(String[] args) {
        Pessoa p = new Pessoa();
        p.idade = -10;  // ❌ dado alterado com valor inválido!
    }
}

Problema: Uma idade negativa não faz sentido no mundo real, mas o programa aceita sem questionar. Isso pode causar erros silenciosos difíceis de encontrar.

A solução: encapsulamento

A solução é proteger o atributo com private e controlar o acesso por métodos. Assim, podemos validar os dados antes de aceitar qualquer alteração: 

          

             Com encapsulamento — solução
            

              

              

              

            

          

          public class Pessoa {
    private int idade;  // ✅ atributo protegido

    public void setIdade(int idade) {
        if (idade >= 0) {          // validação antes de aceitar
            this.idade = idade;
        }
    }

    public int getIdade() {
        return idade;
    }
}

public class Main {
    public static void main(String[] args) {
        Pessoa p = new Pessoa();
        p.setIdade(-10);  // ❌ valor ignorado — validação barrou
    }
}

Resultado: O valor -10 é simplesmente ignorado pelo método, pois a condição idade >= 0 não é satisfeita. O atributo permanece no valor padrão (0).

A analogia do caixa eletrônico

Pense no encapsulamento como um caixa eletrônico (ATM):

Caixa Eletrônico (ATM)
Analogia para encapsulamento
Você não acessa o dinheiro diretamente Não há como pegar notas sem usar a interface — assim como atributos private não podem ser acessados de fora.
Você usa uma interface (botões) Os botões são como os métodos get e set — a única forma de interagir com os dados internos.
O sistema valida tudo internamente Saldo insuficiente? Senha errada? O ATM verifica antes de agir — assim como o setter pode validar antes de alterar.

Encapsulamento é exatamente isso: você não mexe diretamente nos dados — usa "botões" (métodos) para interagir com eles.

Getters e Setters

São métodos usados para acessar e modificar atributos privados. Eles são a base prática do encapsulamento, pois atributos não devem ser acessados diretamente de fora da classe.

📤

Getter

Método de leitura. Retorna o valor de um atributo privado.

📥

Setter

Método de escrita. Altera o valor de um atributo privado.

GET
Getter — método de acesso
Responsável por retornar o valor de um atributo

Padrão

              

                 Estrutura do getter
                

                  

                  

                  

                

              

              public Tipo getNomeAtributo() {
    return atributo;
}

Exemplo

              

                 Getter para o atributo nome
                

                  

                  

                  

                

              

              public class Pessoa {
    private String nome;

    public String getNome() {
        return nome;
    }
}

O atributo nome é privado, então só pode ser lido de fora da classe pelo método getNome(). Sem esse método, o acesso seria impossível.

SET
Setter — método modificador
Responsável por alterar o valor de um atributo

Padrão

              

                 Estrutura do setter
                

                  

                  

                  

                

              

              public void setNomeAtributo(Tipo valor) {
    this.atributo = valor;
}

Exemplo

              

                 Setter para o atributo nome
                

                  

                  

                  

                

              

              public class Pessoa {
    private String nome;

    public void setNome(String nome) {
        this.nome = nome;  // this.nome = atributo da classe
                              // nome      = parâmetro recebido
    }
}

this.nome se refere ao atributo da classe, enquanto nome (sem this) é o parâmetro recebido pelo método. Lembra do this da Unidade 01? É o mesmo conceito aplicado aqui.

Exemplo Completo

Veja a classe Pessoa com dois atributos privados, seus getters e setters — incluindo uma validação no setter de idade: 

          

             Pessoa.java — classe com encapsulamento completo
            

              

              

              

            

          

          public class Pessoa {
    private String nome;
    private int    idade;

    // ── Getter ──────────────────────────────────────
    public String getNome() {
        return nome;
    }

    // ── Setter ──────────────────────────────────────
    public void setNome(String nome) {
        this.nome = nome;
    }

    // ── Getter ──────────────────────────────────────
    public int getIdade() {
        return idade;
    }

    // ── Setter com validação ─────────────────────────
    public void setIdade(int idade) {
        if (idade >= 0) {  // só aceita idade válida
            this.idade = idade;
        }
    }
}

Uso do Encapsulamento na Prática

Veja como usar a classe Pessoa em um programa principal: 

          

             Main.java — usando getters e setters
            

              

              

              

            

          

          public class Main {
    public static void main(String[] args) {

        Pessoa p = new Pessoa();

        p.setNome("Lucas");   // define o nome
        p.setIdade(20);     // define a idade (válida)

        System.out.println(p.getNome());   // lê o nome
        System.out.println(p.getIdade());  // lê a idade

    }
}

Saída esperada no console:

Lucas
20

Comparando: com e sem encapsulamento

❌ Sem encapsulamento
              

                
                

                  

                  

                  

                

              

              Pessoa p = new Pessoa();

// Acesso direto — perigoso!
p.idade = -5;
p.nome = "";
// Nenhuma validação!
✅ Com encapsulamento
              

                
                

                  

                  

                  

                

              

              Pessoa p = new Pessoa();

// Acesso controlado — seguro!
p.setIdade(-5);  // ignorado
p.setNome("Lucas");
// Validação automática!

Por que usar Getters e Setters?

1
Segurança

Evita acesso direto aos atributos com private e controla o que pode ser alterado ou lido.

2
Validação de dados

O setter pode verificar se o valor recebido é válido antes de atribuir ao atributo — evitando dados inconsistentes no objeto.

3
Flexibilidade

Você pode mudar a lógica interna da classe (como a forma de calcular ou armazenar um dado) sem afetar o código que usa a classe.

Validação em detalhe

O setter de idade é um bom exemplo de validação. Veja como ele funciona:

          

             Validação dentro do setter
            

              

              

              

            

          

          public void setIdade(int idade) {
    if (idade >= 0) {        // se a idade for válida (zero ou positiva)...
        this.idade = idade;  // ...atualiza o atributo
    }
    // se não for válida, simplesmente não faz nada
}

Você pode tornar a validação ainda mais robusta — por exemplo, exibindo uma mensagem de erro, lançando uma exceção, ou limitando a idade a um valor máximo. O setter é o lugar ideal para isso.

Convenção de Nomes em Java

Java define uma convenção clara para nomear getters e setters. Seguir essa convenção é importante porque muitas ferramentas e frameworks reconhecem automaticamente esses métodos.

Tipo Padrão Exemplo Retorno
Getter get + NomeAtributo() getNome() Tipo do atributo
Setter set + NomeAtributo(valor) setNome(String nome) void

Caso especial: atributos boolean

Para atributos do tipo boolean, a convenção Java usa o prefixo is no lugar de get: 

          

             Getter para boolean
            

              

              

              

            

          

          public class Conta {
    private boolean ativo;

    // Para boolean: prefixo "is" em vez de "get"
    public boolean isAtivo() {
        return ativo;
    }

    public void setAtivo(boolean ativo) {
        this.ativo = ativo;
    }
}

Resumo da convenção:

  • getNome() — para tipos como String, int, float…
  • isAtivo() — exclusivo para boolean
  • setNome() / setAtivo() — setter sempre usa set

O Construtor Vazio

O construtor vazio (ou padrão) é um construtor sem parâmetros, usado para criar um objeto sem precisar passar valores iniciais no momento da criação. 

          

             Construtor vazio — estrutura
            

              

              

              

            

          

          public class Pessoa {

    private String nome;

    // construtor vazio — sem parâmetros
    public Pessoa() {
    }

}

// Uso:
Pessoa p = new Pessoa();  // chama o construtor vazio

Para que ele serve?

1
Criar objetos "em branco" e definir valores depois

Você cria o objeto primeiro e usa setters para preencher os dados em seguida. Muito comum em sistemas reais.

2
Compatibilidade com frameworks

Muitos frameworks como JPA/Hibernate, serialização JSON/XML e ferramentas de GUI exigem o construtor vazio para criar objetos automaticamente e depois preencher os dados.

3
Inicialização padrão

Você pode usar o construtor vazio para definir valores padrão no momento da criação do objeto.

4
Flexibilidade em testes

Permite criar objetos sem obrigar o preenchimento de todos os dados de uma vez, facilitando testes unitários.

          

             Objeto criado em branco e preenchido depois
            

              

              

              

            

          

          Pessoa p = new Pessoa();   // objeto criado sem dados
p.setNome("Lucas");          // dados definidos depois via setter
        
          

             Construtor vazio com valor padrão
            

              

              

              

            

          

          public Pessoa() {
    nome = "Sem nome";  // valor padrão definido no construtor
}

Comportamento automático do Java

Se você NÃO criar nenhum construtor, o Java cria automaticamente um construtor vazio:

public Pessoa() { }  // criado automaticamente pelo Java

Atenção: se você criar qualquer construtor com parâmetros, o Java não cria mais o construtor vazio automaticamente. Isso vai causar erro:

❌ Causa erro
              

                
                

                  

                  

                  

                

              

              public class Pessoa {
    private String nome;

    // só este construtor existe
    public Pessoa(String nome) {
        this.nome = nome;
    }
}

// ❌ ERRO — construtor vazio não existe!
Pessoa p = new Pessoa();
✅ Funciona
              

                
                

                  

                  

                  

                

              

              public class Pessoa {
    private String nome;

    // construtor vazio — declarado explicitamente
    public Pessoa() { }

    public Pessoa(String nome) {
        this.nome = nome;
    }
}

// ✅ Ambos funcionam!
Pessoa p1 = new Pessoa();
Pessoa p2 = new Pessoa("Lucas");

Quando usar cada tipo?

✔
Use o construtor vazio quando…

Vai usar setters depois, precisa de compatibilidade com frameworks, ou o objeto pode nascer "incompleto" e ser preenchido aos poucos.

✘
Evite o construtor vazio quando…

O objeto não pode existir sem dados válidos. Por exemplo, uma conta bancária sem saldo definido é inconsistente — é melhor forçar a informação via construtor com parâmetro.

          

             Exemplo — quando forçar o construtor com parâmetro
            

              

              

              

            

          

          // ❌ Ruim: conta sem saldo inicial definido é inconsistente
ContaBancaria c = new ContaBancaria();

// ✅ Melhor: forçar o saldo inicial via construtor
public ContaBancaria(double saldoInicial) {
    this.saldo = saldoInicial;
}

Resumo técnico: O construtor vazio serve para permitir a criação de objetos sem inicialização obrigatória, sendo essencial para flexibilidade, frameworks e criação incremental de estado.

Resumo da Unidade

O que você aprendeu nesta unidade:

  • Encapsulamento protege os atributos de uma classe de acessos externos não controlados.
  • Atributos devem ser declarados como private — o nível de acesso mais restritivo.
  • Getters (get + Nome) são métodos para ler o valor de um atributo privado.
  • Setters (set + Nome) são métodos para alterar o valor de um atributo privado.
  • Setters podem incluir validações para garantir a consistência dos dados.
  • Para atributos boolean, o getter usa o prefixo is.
  • O construtor vazio permite criar objetos sem dados iniciais e é exigido por muitos frameworks.
  • Se você criar qualquer construtor com parâmetros, o Java não gera mais o construtor vazio automaticamente.
  • Encapsulamento torna o código mais seguro, organizado e fácil de manter.
